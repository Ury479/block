#### RAG 知识库部分（Retrieve augment generate）
1. RAG：给AI提前引入特定的知识，每次回答问题的时候调用，确保 AI 回答问题不是凭空想象,联想：信息检索 se 专业课
2. 向量数据库：向量体现数据之间的关联度，实现文本编码和语义解析，向量是**语义搜索（Semantic Search）** 的基础
3. 大模型增强提示词负责上下文处理，涉及文本切片
4. Embedding 负责将高维文本/图片等离散数据转化为大模型可理解的向量数据，类比：心智模型，语义编码
5. scope 阈值过高可能会过滤掉希望检索的内容
6. RAG 需要原始文档尽量结构化，md 格式，涉及 GEO 联动
7. VectorStore 就是中间那个"知识库存储层"，文本解析就是文档转向量
8. RAG 实现过程中涉及抽象和面向接口编程，需要抽取可复用的结构，降低系统偶合度，**面向切面编程**，需要开启日志功能
9. 阿里云百炼/Dify 等云知识库能够简化 RAG 开发过程，降本增效
10. Rank 的作用在于**精排序**，让检索出的文档质量更高
11. 文档标签能够提升语义检索效率，根据标签进行检索。**思考：** 黑谷科技实现原理，爬虫检索评论区用户列表，对用户进行贴标签，根据标签进行汇总排序，AI 优化信息抓取效率和销冠话术，作为中台进行系统性集成
12. 标签可作为过滤条件进行参考，metadata 能够增强阿里云百炼/Dify 的检索效率
13. 排序算法只是排列方式，是 Rank 的基础
14. 大模型输出不稳定，导致需要人工介入调整文本切分，靠工程化最大限度降低大模型混乱
15. ETL(Extract Transform Load) 是Embedding,Rank 的前置步骤，是 RAG 的关键一环，数据准备本质是 ETL
16. 提示词工程优化的目的是为了降低大模型幻觉

##### 实现检索增强顾问 Advisor
- 面对复杂输入，先抽象统一模型；面对复杂流程，拆成可插拔流水线；面对多种实现，依赖接口而不是依赖具体类
- DDD:面对复杂业务，先找到领域中的"核心抽象"，再围绕这个抽象建立边界。
- 使用 Soup 读取网页的库
- ETS 依靠`StringUtils.hasText`进行轻量清洗， 避免把空字段写进 metadata，理解防御式编程思想，实现 DocumentReader 预处理
源码实现：让 RAG 检索时，不是检索一整篇文章，而是检索最相关的一小段，同时还能知道这一小段来自哪篇原文。
```java
for (int i = 0; i < texts.size(); i++) {
    String text = texts.get(i);
    //文件列表
    Map<String, Object> metadata = metadataList.get(i);

    List<String> chunks = splitText(text);

    for (String chunk : chunks) {
	    //复制 metadata，过滤 document
        Map<String, Object> metadataCopy = metadata.entrySet()
            .stream()
            .filter(e -> e.getKey() != null && e.getValue() != null)
            .collect(...);

        Document newDoc = new Document(chunk, metadataCopy);

		//接上，进行格式转化，transformer 模式
        if (this.copyContentFormatter) {
            newDoc.setContentFormatter(formatters.get(i));
        }

        documents.add(newDoc);
    }
}
```
