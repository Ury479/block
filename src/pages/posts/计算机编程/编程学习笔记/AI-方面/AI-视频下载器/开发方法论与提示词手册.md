> 核心立场：**站在前人肩膀上，让 AI 出结果，我坐享其成；先闭环，再扩展。**

---

## 一、核心心法（先立纲领）

四条不可让步的总原则，所有决策都拿这四条回溯校验：

1. **拒绝重复造轮子** —— 开发阶段在前人的肩膀上进行，理解开源项目后封装或直接二开。
2. **让 AI 出结果** —— 对标竞品，让 AI 逆向功能实现、直接给结果；卡壳时人工介入给新思路，再交还 AI。
3. **先闭环再扩展** —— AI 罗列任务不贪心，优先聚焦核心功能，自主修复所有报错，跑通核心业务流程后再开新需求。
4. **免费也要为付费设计** —— 即便是免费项目，UI 也要尽可能吸引付费，全程用户思维。

---

## 二、产品策略（定位与变现）

| 维度 | 决策 |
| --- | --- |
| 后端/数据库 | 前期轻后端、轻数据库，后续再接入 |
| UI 来源 | 照抄 Screenshot-to-code / zcode / codex 的前端界面 |
| 变现模式 | 照搬对标产品的付费模式，外网发布赚钱 |
| 调研手段 | 升级 UI 时启用 firecrawl MCP + context7 调研外网现有产品，罗列 md 清单；用 browser-use 登录借鉴 UI、分析付费模式 |
| 图片生成 | 考虑 Gemini nano banana |

**产品策略一句话**：轻后端起步 → 抄界面与付费模式 → 外网变现 → AI 调研迭代 UI。

---

## 三、开发流程（用 AI 把功能做出来）

按顺序执行，每一步都对应下方第六节的提示词：

```
① 竞品逆向      →  ② 开源二开      →  ③ Spec/SDD 沉淀
   (调研对标)       (理解开源代码)      (先设计再开发)
        ↓               ↓                 ↓
④ 核心功能闭环  →  ⑤ 报错沉淀入 AGENTS.md  →  ⑥ 阶段总结 + git + 上下文交接
   (聚焦跑通)        (知识不流失)            (进入下一轮)
```

流程要点（源自提示词思路）：
- **必须先设计再开发**：Agent 走 Spec → SDD 路线。
- **包含人工验证与测试环节**：让 AI 先完成核心功能并经人工验证后再上线。
- **沉淀文档**：让 AI 沉淀需求分析文档和方案设计文档。
- **概念先行**：知道相应概念才能针对性改掉相应 bug。
- **AGENTS.md 是全局记忆**：精简核心，每次报错解决后沉淀进去。

---

## 四、质量与架构约束（限制条件 · 不可破）

这是项目的硬性门禁，每开发一个新功能都要过一遍：

1. **开闭原则** —— 核心功能闭环后，封装为可复用模块，开放 API 接口，**只添加新功能，不修改模块内代码**。
2. **设计模式减重复** —— 使用门面模式等精简重复代码，确保**高内聚、低耦合**。
3. **回归测试门禁** —— 每开发新功能均进行回归测试，自主修复所有报错，**跑通核心业务流程再添加功能**。
4. **UI 付费化升级** —— 前端 UI 升级后改成更易吸引付费的样式（调研流程见第二节）。

---

## 五、Agent 环境配置

- **关掉不必要的 MCP**，减少噪声与 token 消耗。
- **终端选择**：Cursor / Windows 终端指定 Ghostty，或 Git 自带 Linux 终端（Git Bash）。
- **Spec/SDD 优先**：Agent 先设计、后开发。
- **图片生成**：Gemini nano banana。

## 六、可复用 AI 提示词（核心资产）

> 以下模板从原"提示词思路"抽取，已参数化为可填空、可直接粘贴的形式。
> 占位符统一用 `{{ }}` 表示，按项目实际替换即可。

### 6.1 竞品逆向提示词（调研对标）

```text
角色：你是产品 + 前端工程师。
任务：对标竞品 {{竞品名称/网址}}，逆向其功能与变现方式。

执行步骤：
1. 启用 browser-use 访问 {{对标网址}}，理解其 UI 设计与交互。
2. 启用 firecrawl MCP / context7 调研外网同类产品，输出 md 清单。
3. 拆解其核心功能模块与付费墙设计（免费额度、付费点、定价档位）。
4. 给出可直接照搬到我项目的：UI 布局方案 + 付费模式建议。

输出：
- 功能清单 md（含优先级）
- UI 借鉴要点
- 付费模式建议表
```

### 6.2 开源二开提示词（站在前人肩膀上）

```text
角色：你是资深工程师。
任务：在开源项目 {{仓库地址}} 的基础上进行封装或二次开发，拒绝重复造轮子。

执行步骤：
1. 先通读并理解该项目架构、核心模块、扩展点。
2. 评估「直接封装」还是「二次开发」更优，给出结论与理由。
3. 不要修改其核心源码，通过门面模式 / API 接口在外层封装我需要的能力。
4. 列出二开清单（仅新增，不改内码），标注每项依赖的原始模块。

输出：二开方案 + 模块依赖图 + 新增代码骨架。
```

### 6.3 Spec / SDD 沉淀提示词（先设计再开发）

```text
角色：你是架构师。
任务：对功能 {{功能名}} 先产出设计文档，再开发。遵循 Spec → SDD 路线。

必须沉淀两份文档：
1. 需求分析文档（Spec）：背景、目标、范围、非目标、验收标准。
2. 方案设计文档（SDD）：架构、模块划分、接口定义、数据流、风险。

约束：
- 文档写入 docs/ 目录，命名清晰。
- 设计经我确认后，才进入编码。

输出：Spec 文档 + SDD 文档路径。
```

### 6.4 核心功能闭环提示词（聚焦 · 自主修复 · 跑通）

```text
角色：你是执行工程师。
任务：本轮只聚焦核心功能 {{核心功能描述}}，不要贪心加新需求。

纪律：
1. 罗列任务只围绕核心闭环，砍掉一切非必要项。
2. 遇到报错自主修复，直到核心业务流程全程跑通。
3. 完成后必须包含人工验证与测试环节，通过后再算完成。
4. 概念先行：遇到不熟悉的 bug，先讲清涉及的概念，再针对性修复。

完成定义（DoD）：
- 核心流程端到端跑通
- 所有报错已自主修复
- 人工验证通过
```

### 6.5 报错沉淀提示词（写入 AGENTS.md · 全局记忆）

```text
角色：你是项目记忆维护者。
触发：刚刚解决了一个报错 / 踩了一个坑。

任务：把本次解决方案沉淀进 AGENTS.md，确保下次不再踩同一个坑。

写入要求：
- 精简核心，一句话讲清「症状 + 根因 + 解法」。
- 不写废话、不堆日志原文。
- 归类到合适的章节（如「已知坑」「环境配置」「架构约定」）。

输出：本次新增到 AGENTS.md 的条目内容。
```

### 6.6 阶段总结与上下文交接提示词（进入下一轮）

```text
角色：你是项目交接人。
触发：核心功能实现完成后。

任务：整理总结文档，确保进入下一轮开发所需的信息源完整。

执行步骤：
1. 整理本阶段总结文档（做了什么、踩了什么坑、当前状态、下一步建议）。
2. 提交 git，写清晰的 commit message。
3. 把关键信息回写进 AGENTS.md / docs/，让下一轮 AI 能读懂前置项目背景。

目标：换一个新 AI 会话，仅凭仓库 + AGENTS.md + docs/ 就能无缝接手。
```

## 七、每轮开发检查清单（Checklist）

每开一轮新功能，逐项打勾：

- [ ] 是否对标了竞品 / 借鉴了开源？（不重复造轮子）
- [ ] 是否先写了 Spec + SDD 再开发？（先设计）
- [ ] 本轮任务是否只聚焦核心闭环？（不贪心）
- [ ] 是否包含人工验证与测试？（不裸上）
- [ ] 核心流程是否端到端跑通、报错自主修复？（DoD 达标）
- [ ] 是否遵守开闭原则、门面模式减重复？（架构约束）
- [ ] 新功能是否过了回归测试门禁？（质量门禁）
- [ ] 报错/坑是否沉淀进 AGENTS.md？（知识不流失）
- [ ] 是否整理总结 + git 提交 + 上下文交接？（可接手）
- [ ] UI 是否朝「更易吸引付费」方向升级？（变现导向）
