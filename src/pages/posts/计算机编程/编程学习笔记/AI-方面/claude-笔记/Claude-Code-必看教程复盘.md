# Claude Code 必看教程复盘

> 来源：图片课程大纲整理
> 板块：AI 学习 / claude 笔记
> 复盘目标：把 Claude Code 从“会用工具”升级成“会指挥 AI 做工程”。

---

# 一、一句话讲清楚

Claude Code 不是单纯让 AI 写几行代码，而是一个围绕“提示词、计划、Git、架构拆分、前端组件、上下文管理、方案验收”构建的 AI 工程协作系统。

真正要学的不是某个命令，而是一套使用 AI 搭长技术栈的工作流。

---

# 二、核心定位

Claude Code 的价值在于：让 AI 进入真实项目现场，帮我理解需求、拆任务、读代码、改文件、跑验证、总结经验。

但是它不是自动正确的。

我要做的事情是：

- 给它清晰的全局提示词。
- 让它先进入 Plan Mode。
- 用 Git 管理每一步变化。
- 控制上下文输入质量。
- 把复杂文件和复杂业务拆开。
- 用架构图和检查清单验收 AI 方案。

这套方法的本质是：我不是把项目丢给 AI，而是把 AI 纳入一个可控的工程流程。

---

# 三、Claude Code 实战工作流

## 1. 先配置全局提示词

全局提示词决定 Claude Code 默认怎么理解我、怎么写代码、怎么验收任务。

我应该把下面这些内容沉淀成固定规则：

- 先读项目结构，再提出方案。
- 重要任务先 Plan Mode，不要直接改。
- 修改前说明要改哪些文件。
- 修改后必须给出验证方式。
- 不确定的地方要标注假设。
- 不要引入项目没有使用的新技术栈。
- 前端尽量使用已有组件体系。
- 大文件、大模块、大业务逻辑要主动建议拆分。

全局提示词不是装饰，它相当于给 AI 建立工作纪律。

## 2. 每次对话优先使用 Plan Mode

Plan Mode 的作用不是让 AI “多想一会儿”，而是避免它直接冲进代码里乱改。

适合强制使用 Plan Mode 的场景：

- 新功能开发。
- 架构调整。
- 数据库改动。
- 多文件联动修改。
- 前后端联调。
- 不熟悉的项目。
- 需求还没有完全想清楚。

Plan Mode 最少要产出三样东西：

1. 当前项目结构理解。
2. 准备修改的文件和原因。
3. 验收标准和风险点。

如果计划都讲不清楚，说明还不到动手的时候。

## 3. 用 Git 管理项目变化

Claude Code 改项目时，必须配合 Git 使用。

核心原则：

- 每个阶段改动前先确认当前状态。
- 小步提交，不要一次堆太多变化。
- 功能开发、修复、重构分开提交。
- AI 改完后看 diff，再决定是否继续。
- 出问题时能快速定位是哪一步引入的。

Git 的意义不是“保存代码”，而是给 AI 协作加上可追踪、可回滚、可复盘的边界。

## 4. 前端优先使用 v0 画组件

前端页面不要一上来就让 Claude Code 从零手写。

更好的方式是：

1. 先用 v0 生成页面或组件草稿。
2. 让 Claude Code 读取现有项目结构。
3. 把 v0 组件改造成项目内可用版本。
4. 对齐路由、状态管理、接口和样式规范。
5. 再做响应式、交互状态、异常状态补齐。

v0 负责快速出视觉和组件雏形，Claude Code 负责工程化接入和项目适配。

# 四、AI 搭长技术栈的方法

## 1. 使用 opus + ultrathink + plan mode

复杂任务不要省模型能力。

当任务涉及架构、复杂业务、多服务联动、旧项目理解时，优先使用更强模型和更深思考模式。

推荐组合：

- opus：负责高质量推理和复杂代码理解。
- ultrathink：负责深度分析和方案比较。
- plan mode：负责先规划再执行。

这不是为了“显得高级”，而是因为复杂工程任务最怕低质量开局。前面想错了，后面改得越快，错得越远。

## 2. 先用 Roo Code 分析架构

如果项目结构复杂，可以先让 Roo Code 或类似工具做架构分析。

目标不是马上改代码，而是拿到：

- 项目模块划分。
- 主要数据流。
- 前后端调用关系。
- 核心页面和服务。
- 技术栈组成。
- 潜在复杂点。
- 架构图或模块图。

拿到架构图后，再把它发给 Claude Code，让 Claude Code 在清楚上下文的基础上做修改。

这一步能减少 AI 对项目的误判。

## 3. 少发图片，多发精准文本

给 AI 上下文时，能用文字就尽量用文字。

图片适合传达视觉信息，但不适合承载大量工程细节。

如果必须发截图，要注意：

- 截图范围越精准越好。
- 不要把无关窗口、聊天记录、菜单栏一起截进去。
- 配合文字说明：这张图要看哪里、问题是什么、期望结果是什么。
- 对报错信息，优先复制文本，而不是只发截图。

AI 的上下文不是越多越好，而是越准越好。

# 五、复杂度拆分原则

## 1. 单文件行数太多，要主动重构拆分

如果一个文件行数太多，Claude Code 很容易在上下文中丢细节，也容易把不同职责混在一起。

遇到这种情况要主动要求：

- 先分析文件职责。
- 再拆分组件、服务、工具函数或配置。
- 保持外部调用方式稳定。
- 每拆一步都跑验证。

拆分的目的不是追求“好看”，而是降低 AI 和人类都容易出错的认知负担。

## 2. 单块业务逻辑太复杂，要拆成服务

如果一个业务模块同时处理接口、状态、权限、校验、计算、渲染，就应该拆。

推荐拆法：

- 页面只负责展示和交互。
- service 负责请求和业务动作。
- utils 负责纯计算。
- config 负责固定配置。
- types/schema 负责数据结构。
- hooks/store 负责状态管理。

复杂业务不要堆在一个页面文件里。堆得越多，AI 越容易局部修好、整体弄坏。

---

# 六、如何判断 AI 给的方案好不好

判断 AI 方案，不能只看它说得顺不顺。

要从下面几个角度验收：

- 是否符合现有项目技术栈。
- 是否改动范围足够小。
- 是否说明了会改哪些文件。
- 是否保留现有业务行为。
- 是否考虑异常状态和边界条件。
- 是否能运行、能测试、能回滚。
- 是否有清晰的下一步行动。
- 是否把复杂逻辑拆开，而不是继续堆大文件。

一个好的 AI 方案，应该让我更清楚下一步怎么做，而不是让我更依赖它继续解释。

---

# 七、稳定使用 Claude Code 的环境要求

Claude Code 要稳定工作，模型和镜像服务也很关键。

需要关注：

- 模型是否稳定。
- API 是否可用。
- 网络是否容易超时。
- 镜像服务是否支持 Claude Code 常用模型。
- 是否有备用模型或备用 API。
- 配置是否可复现。

工具链稳定，AI 才能真正变成生产力。否则每次都卡在环境、额度、网络和模型切换上。

---

# 八、我的 Claude Code 使用清单

每次开始任务前检查：

- 是否明确了这次任务的目标。
- 是否让 Claude Code 先读项目结构。
- 是否进入 Plan Mode。
- 是否说明禁止引入无关技术栈。
- 是否准备好用 Git 看 diff。
- 是否把截图换成了可复制文本。
- 是否把复杂文件和复杂业务列为风险点。

每次修改完成后检查：

- 是否知道改了哪些文件。
- 是否知道为什么这样改。
- 是否运行了测试或构建。
- 是否有截图、日志、命令输出等验收证据。
- 是否把踩坑沉淀成笔记或 Skill。

---

# 九、复盘结论

Claude Code 的核心不是“让 AI 自动写代码”，而是“建立一套可控的 AI 工程协作流程”。

我真正要练的是三种能力：

1. 提需求：把模糊想法变成明确任务。
2. 管过程：用 Plan Mode、Git、架构图和上下文控制 AI。
3. 做验收：用运行结果、diff、测试和复盘判断方案是否靠谱。

以后使用 Claude Code，不能只追求一次对话解决问题，而要把每次成功经验沉淀成固定提示词、项目规范、服务拆分方法和复盘模板。

---

# 十、下一步行动

1. 整理一份自己的 Claude Code 全局提示词。
2. 给常用项目建立 Git 小步提交习惯。
3. 把 v0 组件接入项目的流程整理成模板。
4. 选一个复杂项目，用 Roo Code 先产出架构图，再交给 Claude Code 修改。
5. 把“AI 方案验收清单”做成每次改代码前后的固定检查项。
