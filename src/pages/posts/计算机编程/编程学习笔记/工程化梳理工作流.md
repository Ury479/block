# 工程化梳理工作流

> 目标：把新的笔记、复盘、日记录内容，转化为能指导后续开发的工程化资产。工程化不是把所有内容变成表格，而是提取“下次可复用的判断、流程、清单、决策和排错经验”。

## 1. 适用范围

本工作流适用于以下原始材料：

| 原始材料 | 常见位置 | 推荐产物 |
|---|---|---|
| 编程日报 | [计算机编程/编程日报](/archive/?q=%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BC%96%E7%A8%8B%2F%E7%BC%96%E7%A8%8B%E6%97%A5%E6%8A%A5) | 开发日报卡、Sprint 任务清单 |
| 项目复盘 | [计算机编程/项目操盘](/archive/?q=%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BC%96%E7%A8%8B%2F%E9%A1%B9%E7%9B%AE%E6%93%8D%E7%9B%98)、项目目录 | 项目复盘报告、项目手册 |
| 报错记录 | [计算机编程/编程错题集](/archive/?q=%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BC%96%E7%A8%8B%2F%E7%BC%96%E7%A8%8B%E9%94%99%E9%A2%98%E9%9B%86) | 错题卡、Debug Playbook |
| 技术选型思考 | 项目笔记、学习笔记 | 技术选型记录、ADR Lite |
| 个人复盘 | [个人管理/成功日记](/archive/?q=%E4%B8%AA%E4%BA%BA%E7%AE%A1%E7%90%86%2F%E6%88%90%E5%8A%9F%E6%97%A5%E8%AE%B0) | 周复盘报告、下周执行合约 |
| 方法论笔记 | 模板、提示词、学习方法 | SOP、检查清单、Prompt 模板 |

## 2. 总流程

```text
输入原始笔记
  ↓
识别笔记类型
  ↓
抽取关键信息
  ↓
选择工程化产物
  ↓
结构化重写
  ↓
生成行动清单
  ↓
回链知识地图与项目笔记
```

## 3. 第一步：识别笔记类型

### 3.1 执行推进型

通常来自编程日报，重点是“今天做了什么、卡在哪里、下一步怎么推进”。

识别特征：
- 有日期。
- 有任务推进记录。
- 有卡点、报错、明日计划。

推荐输出：
- 开发日报卡。
- Sprint 任务清单。
- 阻塞问题清单。

### 3.2 目标复盘型

通常来自周复盘、月复盘、阶段复盘，重点是“目标是否达成、偏差在哪里、下阶段如何调整”。

识别特征：
- 有目标、结果、偏差、反思。
- 包含学习、生活、项目、情绪、执行状态。

推荐输出：
- 周复盘报告。
- 下周执行合约。
- 能力雷达或问题清单。

### 3.3 故障排错型

通常来自错题集、项目报错、环境配置问题。

识别特征：
- 有具体错误现象。
- 有报错信息、截图描述、失败路径。
- 有解决方案或 AI 回答。

推荐输出：
- 标准错题卡。
- Debug Playbook。
- 环境兼容矩阵。

### 3.4 项目指导型

通常来自项目笔记、项目复盘、技术选型记录。

识别特征：
- 涉及需求、架构、技术栈、部署、上线。
- 可指导未来类似项目。

推荐输出：
- 项目手册。
- 项目立项清单。
- 前后端技术选型记录。
- ADR Lite。

## 4. 第二步：抽取关键信息

处理任何原始笔记时，先抽取以下字段：

| 字段 | 提问 |
|---|---|
| 目标 | 这件事原本要解决什么问题？ |
| 动作 | 实际做了哪些操作？ |
| 结果 | 最后完成了什么？有什么证据？ |
| 阻塞 | 卡在哪里？为什么卡？ |
| 决策 | 做过哪些选择？为什么这样选？ |
| 报错 | 出现了什么错误？错误信息是什么？ |
| 根因 | 真实原因是什么？不是表层现象是什么？ |
| 经验 | 下次遇到类似情况如何更快处理？ |
| 下一步 | 之后最小可执行动作是什么？ |

## 5. 第三步：选择输出产物

### 5.1 日报转开发日报卡

适合：日常推进、编程日报、项目进度记录。

输出结构：

```markdown
# 开发日报卡｜YYYY-MM-DD

## 今日结论
一句话说明今天推进结果。

## 今日完成
- [ ] 动作 1
- [ ] 动作 2

## 卡点与原因
| 卡点 | 原因 | 当前状态 | 下一步 |
|---|---|---|---|
|  |  |  |  |

## 证据
- 提交记录：
- 文件路径：
- 截图/运行结果：

## 明日 2 小时首攻点
- [ ] 第一优先级动作

## 可复用经验
- 
```

### 5.2 复盘转复盘报告

适合：周复盘、月复盘、阶段总结。

输出结构：

```markdown
# 复盘报告｜主题｜周期

## 结论一句话

## 目标达成情况
| 目标 | 结果 | 达成度 | 证据 |
|---|---|---|---|
|  |  |  |  |

## 偏差分析
| 偏差 | 原因 | 影响 | 修正动作 |
|---|---|---|---|
|  |  |  |  |

## 下周期执行合约
- [ ] 动作 1
- [ ] 动作 2

## 可复用经验
- 
```

输出结构：

```markdown
# 错题卡｜错误主题

## 错误现象

## 环境信息
- 系统：
- 技术栈：
- 版本：
- 运行命令：

## 报错信息
```text
粘贴关键报错，不需要整段无关日志
```

## 复现路径
1. 
2. 
3. 

## 根因分析

## 解决方案

## 预防清单
- [ ] 

## 关键词
#错题卡 #工程化梳理
```

## 6. 第四步：回链规则

每个工程化产物至少建立 3 类链接：

1. **来源链接**：链接回原始日记、复盘或项目笔记。
2. **主题链接**：链接到对应项目、技术栈或错题分类。
3. **地图链接**：链接到 [000 知识地图](/archive/?q=000%20%E7%9F%A5%E8%AF%86%E5%9C%B0%E5%9B%BE) 或 [计算机编程/000 编程知识地图](/archive/?q=%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BC%96%E7%A8%8B%2F000%20%E7%BC%96%E7%A8%8B%E7%9F%A5%E8%AF%86%E5%9C%B0%E5%9B%BE)。

推荐标签：

- `#工程化梳理`
- `#项目复盘`
- `#错题卡`
- `#ADR`
- `#日报转化`
- `#技术选型`

## 7. 第五步：质量检查

工程化整理完成后，用以下清单自检：

- [ ] 是否保留了原始事实，没有过度脑补？
- [ ] 是否提炼出一句话结论？
- [ ] 是否有可执行的下一步？
- [ ] 是否能指导未来类似开发？
- [ ] 是否链接回原始笔记？
- [ ] 是否链接到知识地图或项目地图？
- [ ] 是否避免把私人情绪全部任务化？
- [ ] 是否区分了一次性问题和长期决策？

## 8. 不要过度工程化的内容

以下内容不建议强行转成工程文档：

- 纯情绪宣泄。
- 一次性、不可复用的小问题。
- 尚未验证的灵感。
- 没有事实支撑的判断。
- 私人生活细节。

处理方式：保留原始笔记，只提取其中可复用的行动、判断或方法。

## 10. 使用提示词

当你给我新的笔记、复盘或日记录时，可以这样说：

> 请按 `工程化梳理工作流`，把这篇笔记整理成能指导后续开发的文档。要求保留原始事实，提炼工程决策、行动清单、风险和可复用经验，不要过度改写私人表达。

如果是项目类内容，可以补充：

> 请同步生成项目立项清单、前后端技术选型记录、项目复盘模板中的对应内容。
