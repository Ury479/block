# 鱼皮命令行首页项目｜项目开发指导文档

> 目标：把本项目笔记整理成一份可复用的工程化开发指南。后续做前端项目、命令行工具首页、效率工具、个人工作台或类似产品时，可以直接参考本文完成需求判断、技术选型、架构取舍和开发检查。

## 1. 项目定位

### 1.1 项目名称

鱼皮命令行首页项目，也可理解为“极客范浏览器首页 / 命令行式个人工作台”。

### 1.2 核心目标

构建一个面向开发者或重度电脑用户的效率型首页，用命令行交互或快捷操作替代传统多页面跳转，提升搜索、导航、账号同步和常用工具访问效率。

### 1.3 主要价值

- 减少多页面切换成本。
- 集中管理常用入口和工具。
- 支持个人高频工作流沉淀。
- 为后续浏览器首页、命令面板、AI 工作台等项目提供技术验证基础。

### 1.4 目标用户

| 用户类型 | 需求特征 | 设计启发 |
|---|---|---|
| 自己 | 高频搜索、页面切换、多端同步、效率提升 | 以自身真实痛点作为第一需求来源 |
| 开发者 | 偏好快捷键、命令行、可配置入口 | 强调键盘操作和可扩展性 |
| 身边用户 | 可能不熟悉命令行，但需要快速访问工具 | 需要兼顾易用性和学习成本 |

## 2. 需求分析流程

### 2.1 需求来源

需求不能只来自主观想象，需要经过多来源验证。

1. **自身需求**
   - 是否真的存在使用痛点？
   - 是否每天或每周高频出现？
   - 是否能明显节省时间或减少操作？

2. **全网搜索**
   - 在 GitHub、掘金、StackOverflow、npm、Openbase 等平台检索类似项目。
   - 查找竞品、开源实现、技术方案和用户反馈。
   - 通过外部项目验证自己的需求是否有普遍性。

3. **竞品逆向**
   - 分析竞品做了哪些功能。
   - 还原作者的产品思路和技术取舍。
   - 判断哪些功能值得借鉴，哪些功能不适合当前阶段。

4. **身边人验证**
   - 以自己为主需求来源，以他人反馈作为参照。
   - 不要为了迎合他人加入大量低频功能。

### 2.2 需求优先级判断

每个功能进入开发前，都要先判断是否值得做。

| 判断问题 | 决策 |
|---|---|
| 是否影响核心使用链路？ | 影响则优先做 |
| 是否只是锦上添花？ | 延后 |
| 是否当前能力或时间明显不支持？ | 先砍掉或找替代方案 |
| 是否已有成熟轮子？ | 优先调研复用 |
| 是否会显著增加后端、安全或维护成本？ | 谨慎加入 |

## 3. 竞品与开源项目调研方法

### 3.1 检索渠道

| 渠道            | 用途                       |
| ------------- | ------------------------ |
| GitHub        | 查找完整项目、代码结构、README、Issue |
| npm           | 查找前端库、命令行库、组件库           |
| Openbase      | 对比 npm 包质量、维护情况、下载量      |
| 掘金 / 博客       | 查找中文实践教程和项目复盘            |
| StackOverflow | 查找具体报错和技术边界问题            |

### 3.2 调研重点

调研时不要只看“项目能不能跑”，更要看它是否适合自己复用。

- 项目是否近期维护？
- README 是否清晰？
- 是否有在线 Demo？
- Issue 中是否有大量未解决问题？
- 技术栈是否和自己当前能力匹配？
- 项目是重前端、重后端，还是前后端强耦合？
- 如果出错，自己是否有能力定位和修复？

### 3.3 轮子复用原则

| 场景 | 建议 |
|---|---|
| 学习阶段 | 可以参考轮子思路，但尽量亲手实现关键功能 |
| 技术沉淀期 | 多试几个方案，比较成本和稳定性 |
| 企业开发 / 交付阶段 | 优先选择成熟方案，减少自研成本 |
| 轮子文档差、报错多、调不通 | 放弃复用，改为自研或换方案 |

## 4. 技术选型原则

### 4.1 选型基本流程

1. 明确业务流程。
2. 拆分功能模块。
3. 找已有实现和成熟库。
4. 写 Demo 验证关键功能。
5. 评估学习成本、维护成本和报错风险。
6. 再决定是否正式引入项目。

### 4.2 选型判断标准

| 标准 | 说明 |
|---|---|
| 实现速度 | 是否能快速搭建 MVP |
| 学习成本 | 当前是否能理解和维护 |
| 社区活跃度 | 是否持续维护、有教程、有问题解答 |
| 报错可控性 | 出现问题后能否定位 |
| 扩展能力 | 后续是否支持账号、同步、配置、插件等能力 |
| 部署成本 | 是否容易部署和持续维护 |

### 4.3 阶段化选型策略

#### 学习阶段

- 目标是理解原理和流程。
- 可以借鉴开源项目，但不要完全复制。
- 关键模块尽量自己实现一遍。

#### 技术沉淀阶段

- 目标是比较多个技术方案。
- 可以多试 npm 包、组件库、框架方案。
- 记录每种方案的优缺点和报错点。

#### 项目交付阶段

- 目标是稳定上线。
- 优先成熟方案，减少不必要的自研。
- 能不造轮子就不造轮子，但必须保证自己能维护。

## 5. 开发方法论 @

### 5.1 敏捷开发与稳定开发

| 开发方式 | 适用场景 | 核心原则 |
|---|---|---|
| 敏捷开发 | 小功能、MVP、个人项目、快速验证 | 重效率、重实现、重反馈 |
| 稳定开发 | 支付、银行、安全敏感、核心系统 | 重安全、重测试、重可靠性 |

本项目更适合采用敏捷开发：先完成可用版本，再逐步补齐体验、同步、配置和扩展能力。

### 5.2 敏捷开发决策规则

| 问题 | 决策 |
|---|---|
| 这个功能是否影响上线？ | 影响则做 |
| 不做是否影响核心体验？ | 不影响则延后 |
| 是否只是“看起来很高级”？ | 先不做 |
| 是否能通过文案或占位先解释？ | 可以先画大饼，后续迭代 |
| 是否需要复杂后端能力？ | 先判断是否能前端化或本地化实现 |

## 6. 架构取舍

### 6.1 重前端方案

适合本项目的早期版本。

**优点：**
- 部署简单。
- 后端攻击面小。
- 适合快速上线和个人项目验证。
- 前端同学更容易理解和维护。

**缺点：**
- 数据同步能力有限。
- 用户体系、权限、持久化能力较弱。
- 复杂协作功能需要后端支持。

**适用功能：**
- 本地命令配置。
- 快捷导航。
- 搜索入口聚合。
- 主题配置。
- 静态页面展示。

### 6.2 重后端方案

适合需要账号、多端同步、团队协作、数据持久化的版本。

**优点：**
- 能支持登录、同步、用户配置和数据管理。
- 适合长期产品化。
- 数据结构更规范。

**缺点：**
- 后端存在被攻击风险，需要安全设计。
- 部署和维护成本更高。
- 需要处理接口、数据库、鉴权、缓存等问题。

**可选技术：**
- Java + Spring Boot。
- MyBatis / ORM 数据访问层。
- Redis 做分布式登录或缓存。

### 6.3 前后端一体方案

例如 Next.js 或类似全栈框架。

**优点：**
- 前后端代码组织更集中。
- 更适合全栈开发者。
- 可以兼顾页面和接口。

**缺点：**
- 对初学者心智负担更重。
- 如果项目边界不清晰，容易前后端耦合。

### 6.4 当前推荐架构

早期建议采用：

```text
Vue 前端页面
  ├─ Vue Router：页面切换
  ├─ Pinia：状态管理
  ├─ axios：接口请求
  ├─ Ant Design Vue：组件库
  └─ Vercel：前端部署
```

如果后续加入登录和多端同步，再扩展：

```text
Vue 前端
  ↓ axios
Spring Boot / Node.js 后端
  ↓ ORM / MyBatis
数据库
  ↓ 缓存或会话
Redis
```

## 7. 技术栈说明

### 7.1 前端技术栈

| 技术 | 作用 | 常见问题定位 |
|---|---|---|
| Vue | 前端开发框架，用于组件化页面 | 页面渲染、组件状态异常 |
| Vue Router | 页面路由和页面切换 | 页面跳转失败、路径不匹配 |
| Pinia | 状态管理，对标 Vuex | 状态丢失、页面数据不同步 |
| axios | 请求后端接口 | 404、500、跨域、接口参数错误 |
| Ant Design Vue | UI 组件库 | 样式覆盖、组件属性配置 |
| TypeScript | 类型约束，提高长期维护效率 | 类型定义错误、接口类型不匹配 |

### 7.2 后端技术栈

| 技术 | 作用 | 使用条件 |
|---|---|---|
| Spring Boot | Java 后端服务开发 | 需要稳定后端、用户系统、接口层 |
| MyBatis / ORM | 数据库访问层 | 需要结构化数据库操作 |
| Redis | 分布式登录、缓存、会话管理 | 需要登录态、缓存或高频读取 |
| Maven | Java 项目依赖管理 | Spring Boot 项目标准工具 |

### 7.3 部署技术栈

| 平台 | 适用对象 | 特点 |
|---|---|---|
| Vercel | 前端项目 | 免运维、部署快、适合静态页面和前端应用 |
| 微信云托管 | 后端服务 | 类容器化部署，降低服务器配置成本 |
| CI/CD yml | 自动化部署 | Vercel 可兼容，但早期可不必手写复杂流程 |

## 8. 报错定位规则

### 8.1 前端报错定位

| 报错现象 | 优先检查 |
|---|---|
| 页面无法切换 | Vue Router 配置、路径、组件导入 |
| 页面状态异常 | Pinia store、状态初始化、持久化逻辑 |
| 控制台接口红色报错 | axios 请求地址、参数、后端状态码 |
| 404 | 接口路径或路由路径错误 |
| 500 | 后端服务异常或数据库异常 |
| 页面拿不到数据 | API 返回结构、字段名、异步加载逻辑 |
| TS 报错 | 类型定义、接口字段、组件 props |

### 8.2 技术方案失控判断

如果某个轮子或技术方案出现以下情况，应考虑停止投入：

- 文档缺失，关键用法无法确认。
- 报错持续超过自己重新实现的预计时间。
- Issue 中存在大量类似未解决问题。
- 项目依赖过旧，和当前技术栈冲突严重。
- 核心逻辑不可控，后续无法维护。

## 10. 后续迭代路线

### 10.1 MVP 阶段

优先完成：

- 首页基础布局。
- 命令输入框。
- 常用网站快捷入口。
- 搜索跳转。
- 本地配置或静态配置。

### 10.2 增强阶段

可以加入：

- 快捷键操作。
- Pinia 管理用户配置。
- 主题切换。
- 历史命令记录。
- 常用命令分组。

### 10.3 产品化阶段

谨慎加入：

- 登录注册。
- 多端同步。
- 云端配置。
- 用户自定义命令。
- 插件系统。
- AI 搜索或 AI 命令助手。

## 11. 可复用经验

1. 项目不要只从“我想做”开始，而要通过搜索和竞品验证需求。
2. 技术选型必须先写 Demo，不能只看教程和宣传。
3. 学习阶段可以造轮子，交付阶段优先用成熟方案。
4. 轮子如果报错不可控，及时止损比硬调更重要。
5. 敏捷开发的核心是先上线核心链路，不要被非核心功能拖住。
6. 前端项目早期尽量降低后端依赖，减少安全和部署复杂度。
7. 当项目需要账号、多端同步、权限和数据持久化时，再认真设计后端。
8. 每次报错都应该沉淀为可复用经验，而不是只解决当下问题。
