# 问兰小程序开发教训｜工程化复盘

> 复盘目标：把这次小程序开发里的混乱、试错和时间损耗，沉淀成下次可直接执行的开发流程。  
> 核心原则：拿到需求不急着开工，先做需求定位、产品对标、技术选型、协作节奏、工具链验证和测试闭环。

## 结论一句话

这次最大的教训不是“代码不会写”，而是开工前没有把需求、对标项目、技术栈、盲区、协作方式和验收标准一次性对齐，导致开发过程中容易陷入方向摇摆、工具不稳定、重复试错和时间损耗。

下次做问兰/微信小程序项目，必须先完成一份 30 分钟版本的《开工判断卡》，确认需求值不值得做、用什么项目参考、自己会什么不会什么、找谁对齐、如何测试，再进入正式开发。

---

## 1. 原始教训提炼

1. **需求定位：使用 GPT 定下来**
   - 不要凭感觉理解需求。
   - 先用 GPT 把需求拆成用户、场景、功能、边界、验收标准。
   - 如果需求一句话讲不清，说明还不能开工。

2. **产品对标：Gemini Pro / GPT 优先检索国内开源项目**
   - 先找国内类似项目、GitHub 项目、即刻/博客/掘金经验。
   - 不要直接从零写。
   - 先判断别人怎么做、有哪些坑、是否有可复用代码。

3. **技术选型：事先整理自己熟悉的技术栈，开会对齐认知差**
   - 先列出自己熟悉的技术栈：原生小程序、云开发、Supabase、Node、Vercel、Docker、AI 工具链等。
   - 再根据需求确定技术栈和盲区。
   - 每 1h 和花海跟进一次，防止方向走偏。

4. **团队协作：主动承担测试开发工作**
   - 不只做页面和功能，也要做测试。
   - 针对同事开发的功能做单元测试、接口测试、回归测试和核心链路验收。
   - 测试不是收尾工作，而是开发中持续做的质量门。

5. **AI 工具：提前配好 Codex、Claude Code、Cursor**
   - 工具链要在开工前确认稳定。
   - 不能边开发边修工具。
   - 配置成功后要整理配置笔记，下一次直接复用。

## 2. 目标达成情况

| 目标 | 实际结果 | 达成度 | 证据/表现 |
|---|---|---|---|
| 明确小程序需求定位 | 只形成了初步方向，缺少用户场景、功能边界和验收标准 | 部分达成 | 需要依赖 GPT/关键人进一步确认 |
| 找到可对标项目 | 已意识到要优先检索国内开源项目，但还没有形成固定检索流程 | 部分达成 | GitHub、即刻、GPT/Gemini 检索应前置 |
| 确定技术选型 | 已意识到要整理熟悉技术栈和盲区，但没有形成选型表 | 部分达成 | 技术栈、部署方式、后端方案需要开工前确认 |
| 建立协作节奏 | 已意识到要每 1h 和花海跟进，但还没有写成制度 | 部分达成 | 超过 30min 无进展要主动求助 |
| 建立测试职责 | 已意识到可以主动承担测试开发，但缺少测试清单 | 部分达成 | 单元、接口、回归、冒烟需要明确 |
| 稳定 AI 工具链 | 已明确 Codex、Claude Code、Cursor 要提前配置 | 部分达成 | 工具配置笔记需要沉淀 |

## 3. 偏差分析

| 偏差 | 根因 | 影响 | 修正动作 |
|---|---|---|---|
| 一拿到任务就想开工 | 缺少开工前判断流程 | 容易做错方向，浪费时间 | 先填《开工判断卡》，再写代码 |
| 需求边界不清 | 没有把需求拆成用户、场景、功能、验收标准 | 开发过程中反复变动 | 用 GPT 先生成需求拆解，再找花海确认 |
| 对标不足 | 没有系统检索国内项目和开源项目 | 容易重复造轮子 | GPT/Gemini/GitHub/即刻四路检索 |
| 技术选型凭感觉 | 没有列出熟悉技术栈和盲区 | 容易踩部署、架构、兼容性坑 | 开工前完成技术选型表 |
| 工具链临时配置 | Codex、Claude Code、Cursor 没有提前验证 | 开发时间被工具问题吞掉 | 每个工具提前跑通一个最小任务 |
| 测试意识后置 | 只关注功能实现，没有同步设计验收 | 交付质量不可控 | 开发中同步写测试清单和回归路径 |
| 求助节点太晚 | 卡住后容易继续硬扛 | 30min 问题拖成 2h/8h | 超过 30min 无明确进展，带日志找花海 |

## 4. 下次开工 SOP

### 4.1 第一步：需求定位

拿到需求后，先问 6 个问题：

1. 用户是谁？
2. 用户要完成什么动作？
3. 这个功能解决什么痛点？
4. 最小可交付版本是什么？
5. 哪些功能本期不做？
6. 怎么证明它做完了？

GPT 提示词：

```text
请帮我把这个微信小程序需求拆成：
1. 用户角色
2. 使用场景
3. MVP 功能
4. 非本期范围
5. 技术风险
6. 验收标准
7. 需要向负责人确认的问题

需求如下：
……
```

### 4.2 第二步：产品对标

检索顺序：

1. GPT Search：先理解行业和功能模式。
2. Gemini Pro：补充搜索结果，交叉验证。
3. GitHub：找可运行开源项目。
4. 即刻/掘金/博客：找国内开发者踩坑记录。
5. 微信开放社区：查小程序平台限制、组件限制、审核限制。

对标时不只看页面，要看：

- 是否是原生微信小程序。
- 是否有后端。
- 是否能本地运行。
- 是否有环境变量。
- 是否支持当前需求。
- 是否有部署/上线说明。
- 是否近期维护。

### 4.3 第三步：技术选型

先整理自己会什么，再判断项目要什么。

| 选型项 | 优先方案 | 备选方案 | 判断标准 |
|---|---|---|---|
| 小程序端 | 原生微信小程序 | 已有项目技术栈 | 不引入 Taro/Uni-app/React/Vue，除非仓库已有 |
| 数据更新 | `this.setData()` 精准路径 | 全量更新 | 减少性能浪费 |
| 页面结构 | `Page` + 可复用 `Component` | 大页面硬写 | 复用组件优先用 `Component()` |
| 列表渲染 | `wx:for` + 稳定 `wx:key` | 无 key | 防止渲染异常 |
| 样式 | WXSS + `rpx` | 随意 px | 保持小程序响应式 |
| 后端 | 现有后端/云函数/Supabase | 临时拼接口 | 看项目已有架构和上线成本 |
| 部署 | 先验证能否跑通 | 直接追求上线 | 本地跑通优先于部署折腾 |
| AI 工具 | Codex + Claude Code + Cursor | 单工具硬扛 | 分工明确，提前验证 |

### 4.4 第四步：协作节奏

每 1h 向花海同步一次，内容控制在 5 行：

```text
当前目标：
已完成：
卡点：
我的判断：
需要你确认：
```

超过 30min 没有明确进展，立即求助。求助时不要只说“我不会”，要带上：

- 当前需求理解。
- 已经尝试的方法。
- 报错截图/日志。
- 自己的初步判断。
- 想让对方帮忙决策的问题。

## 5. 测试开发职责

下次可以主动承担测试开发工作，把自己从“只会写功能”升级成“能保障交付质量”。

### 5.1 小程序最小测试清单

- [ ] 页面能正常打开。
- [ ] 主要按钮能点击。
- [ ] 表单输入、提交、校验正常。
- [ ] 列表渲染有稳定 `wx:key`。
- [ ] tab 页使用 `wx.switchTab`。
- [ ] 普通页面跳转不超过页面栈限制。
- [ ] iOS 日期解析已处理 `-` 转 `/`。
- [ ] 网络请求失败有提示。
- [ ] 登录态失效有处理。
- [ ] 关键功能在微信开发者工具和真机都测过。

### 5.2 回归测试路径

每次同事改完功能，至少跑这 4 条：

1. 首页进入核心功能。
2. 登录/授权/用户信息链路。
3. 关键表单或交易动作。
4. 返回、刷新、异常网络情况下的表现。

### 5.3 测试记录模板

```markdown
## 测试记录｜功能名

- 测试时间：
- 测试人：
- 测试环境：微信开发者工具 / iPhone / Android
- 测试范围：
- 通过项：
- 问题项：
- 复现路径：
- 截图/日志：
- 修复状态：
```

## 6. AI 工具链预案

开工前必须完成工具稳定性检查。

| 工具 | 用途 | 开工前检查 |
|---|---|---|
| GPT / Gemini | 需求拆解、产品对标、资料检索 | 能否稳定搜索，输出是否可验证 |
| Codex | 仓库阅读、代码修改、工程化整理 | 能否读取项目、运行命令、修改文件 |
| Claude Code | 长上下文架构理解、复杂重构 | 能否接入项目、是否有可用额度 |
| Cursor | 局部代码编辑和补全 | 模型、索引、终端是否可用 |
| 微信开发者工具 | 小程序运行和调试 | 项目能否导入、appid/测试号是否准备好 |

工具链原则：

1. 工具不稳定时，不进入正式开发。
2. 每个工具先跑一个最小任务。
3. 配置成功后写成配置笔记。
4. 工具问题超过 30min，先切换方案，不硬扛。

## 9. 可复用经验

1. **需求不清，不开工。**
   - 需求没有用户、场景、MVP、验收标准，就不是开发任务，只是模糊想法。

2. **项目不熟，不部署。**
   - 先看架构、依赖、环境变量、运行方式，再考虑上线。

3. **超过 30min 没进展，就求助。**
   - 求助不是丢人，带着日志和判断求助是工程协作能力。

4. **AI 不是替你判断的人，而是帮你加速验证的工具。**
   - 最终仍然要自己判断需求、技术栈、风险和验收。

5. **测试不是最后一步，而是开发的一部分。**
   - 能写测试、会回归、能发现问题，才算真正参与项目交付。

6. **结构化输出，工程化文档，配上强执行力，是自己计算机路上最大的贵人。**
   - 每次踩坑都要变成模板、清单、SOP、错题卡或项目手册。
