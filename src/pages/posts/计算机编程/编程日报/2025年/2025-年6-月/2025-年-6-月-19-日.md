#### 技术总结
##### 1. mybatis依赖版本出错
```
## Spring Boot 报错：Invalid value type for attribute 'factoryBeanObjectType': java.lang.String
```
###### 兼容性结论

- mybatis-spring-boot-starter 2.2.2 只支持 Spring Boot 2.x，不兼容 Spring Boot 3.x。
- 你必须升级到 3.0.4，否则会出现各种 Bean 注册、注解等兼容性问题（你遇到的 Invalid value type for attribute 'factoryBeanObjectType' 就是典型的版本不兼容问题）。

##### 2.分页-查询语句
```
@Override  
public PageBean page(Integer page, Integer pageSize) {  
    // 设置分页参数  
    PageHelper.startPage(page, pageSize);  
      
    // 执行查询  
    List<Emp> empList = empMapper.list();  
      
    // 获取分页结果  
    com.github.pagehelper.PageInfo<Emp> pageInfo = new com.github.pagehelper.PageInfo<>(empList);  
      
    // 封装结果  
    return new PageBean(pageInfo.getTotal(), pageInfo.getList());  
}
```

9. 职场沟通留下证据，理清做事顺序，不要被人拿捏，遇到事情冷静沟通，争执不下就请上级拍板，
	1. 手上还有别的任务，你这个改动有多少工作量，如果开发这个，另一个需求估计就不太能按时交付，巴拉巴拉，然后再说，先提个需求，到时候看看来不来得及。
	2. 最常见的就是「功能和 xx 保持一致」，没有具体的需求改动点、对接方，以及功能点，就是想当然让开发去做需求。遇到这种情况，第一反应必须是拒绝。不要以为，我能明白他的意思，做成什么样心里有数。没有落实 prd，万一最终实现出来他不验收，或者是开发过程中遇到各种问题导致需求延期，但是你又没有证据证明是需求的问题，那么这口锅就会结结实实的落到你头上。大部分做开发的人都比较单纯，而且也不太擅长与人沟通，但是遇到恶心产品，就会受欺负。当然受一两次欺负也不打紧，属于吃一堑长一智了，但是咱也不能任人拿捏，不能次次都受欺负，所以总要学一点和产品周旋的技巧。
