document.getElementById("toastbtn").onclick = function() {
  // 1. 查找所有 toast 元素
  var toastElList = [].slice.call(document.querySelectorAll('.toast'))
  
  // 2. 创建 Bootstrap Toast 实例
  var toastList = toastElList.map(function(toastEl) {
    return new bootstrap.Toast(toastEl)
  })
  
  // 3. 显示所有 toast
  toastList.forEach(toast => toast.show()) 
}

#### 阅读总结
1. 前端开发动手第一，勿以浮沙筑高台，先完成后完美，在完成的过程中逐步优化
2. 读得懂代码，知道每部分的功能，清楚怎么查文档，就够了
	1. 知道每行代码的意义，能看文档写出来就行
	2. 何为完美？每个人的定义不同。先完成，再根据实际情况不断优化才是王道
	3. 自己的每个项目都要整理，可以记录自己的成长
	4. 每学完一个知识，都去回想、总结、写心得，这个时间的花费是值得的
3. 工程师掉发，说明技术不够，经常熬夜焦虑,正常做法是总结重复报错，整理资源文档，降低查询损耗，学会用他人造好的轮子，而非重复造轮子
	1. 有没有‏种可能，是因为太菜，؜需求做不出来、Bug⁢ 改不完，所以才经常؜熬夜加班、精神压力极‌大，导致头发熬没了呢？
	2. 1. **第 2 点是程序员对排期的错误估算。我发现一个有趣的事情，需求是做不完的，你需求做的越快，新需求来的就越快；而且我们很多时候只考虑了做需求的时间，没有考虑改 Bug 的时间。但现实却有可能是改 Bug 的时间比开发的时间还要长。所以可能的话，还是别把需求排太满，预留一部分时间改 Bug。**

	6. 我觉得衡量优秀程序员的标准绝不止有技术，比如问‏题的解决能力，同一件事，你完成的比别人快比别人好；比如业务理解能力，给你一个需求，很快就能؜判断它是否合理、梳理清楚流程；比如沟通能力，你能够很好地维护用户、组员、同事、跨部门合作者⁢的关系，从杂乱的消息中提取出有效信息；比如产品思维，你能给出更好的建议来推动产品发展；比如؜管理能力，善于组织成员、推动团队发展。再比如分享表达能力，能把自己学会的东西清晰地讲出来、‌让别人也能理解，我觉得很酷的事情，也是我一直做分享的原因。
4. 基础知识的用途就是搭建知识体系，回归第一性原理，迅速归纳拆解梳理脉络，相互联系回顾审查，刻意练习，实现递归收敛，源头发散，三生万物
	1. 基础，并不意味着初级。学基础知识有一个好处，就是能架起一个体系，之后你再学其他东西，能迅速判断他属于这个体系中什么分支，从而能触类旁通，学的很快。这也是你以后能区别于初级程序员的最重要标准之一
	2. 那么，计算机都有哪些相关的基础知识呢？主要有几大块：
		1. 算法以及代码熟练度（可以通过刷 LeetCode，看网上相关解答来达到）；
		2. 代码抽象。就是如何从高层理解编程这件事，比如如何抽象，封装，如何看待程序语言。学学 Berkeley 的 CS61a 就行：[https://inst.eecs.berkeley.edu/~cs61a/fa19/](https://inst.eecs.berkeley.edu/~cs61a/fa19/)
		3. 计算机体系结构。CSAPP 这本书不错。
		4. 操作系统。MIT 6.828 这个公开课做了就行。
		5. 数据库。可以做 CMU 15445。
		6. 计算机网络。这个了解下 TCP/IP 五层协议就行，着重看下 TCP 的一些建立连接过程、拥塞控制算法。
		7. 编译原理、离散数学 这些东西更底层一些，有空可以再补充。
	3. 面试中考察算法题仅仅是一种方式，本质是面试官想透过你的代码和回答，辨别你是否有扎实的计算机基础知识、学习和思考能力、是否会举一反三等等。即使你没有学过“面试岗位”所需要的工作编程语言，若你的计算机基础知识扎实，很多公司愿意提供机会来培养你。
	4. 仅仅从书本上学习知识是不够的，编程就是要不停地实践，制定适当难度的任务，得到有意义的反馈，重复并及时改正错误。不仅要熟悉计算机底层内容，亦要了解技术行业最新知识，终身学习的态度是必要的。如果你是最好的，请着眼你领导项目的能力，尝试用你的想法鼓舞其他人;如果你是最差的，请学习其他人都是怎么做的，了 解为什么要这么做的原因。
